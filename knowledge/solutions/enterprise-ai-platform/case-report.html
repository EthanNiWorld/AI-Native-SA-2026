项目案例 GPU 自建推理 AWS 迁移 2026-05

企业自建 AI 推理平台

某 500 强企业从 AWS H20 GPU 集群迁移至阿里云，构建统一 AI 网关 + 自建推理集群 + 云端 API Fallback 的混合推理平台

客户类型 某 500 强企业
核心产品 灵骏 H20 + ACK + Higress
GPU 规模 H20-3e × 8 台（设计满配）
更新日期 2026-05-18
📋 01  项目背景与客户画像

迁移背景

客户原基于 AWS 的 H20 GPU 集群，因国内监管合规压力及云上成本优化需求，决策迁移至阿里云。核心业务线：

  • ZX App（AI 咨询类）— 高频对话场景
  • Audit App（文档审计类）— 全量合规审计

AWS 原方案痛点

  • GPU 资源与业务调度分离，资源视图割裂
  • 多业务 App 各自对接推理 Endpoint，运维复杂
  • 各层监控散装拼凑，无全链路视图
  • Fallback 策略需业务侧自行实现
  • 合规审计需额外开发，难满足国内监管

阿里云核心优势

  • Higress AI 网关统一入口，全流量管控
  • 从网关到 GPU 卡全链路一体化可观测
  • 全量 Prompt/Response 审计，NAS 持久化合规
  • ACK 统一纳管 ECS + 裸金属 + 百炼 API

预算规模

  • 灵骏裸金属 H20-3e × 8 台（满配设计，分批纳管）
  • NAS 存储：模型文件 1TB（RWX）
  • 监控存储：Grafana 5TB + Loki 10TB + Prometheus 10TB
🎯 02  核心需求分析
优先级 需求描述 对应云产品 / 服务
P0 AI 网关统一入口：一个接入点管控所有 LLM 流量，自建 GPU 与云端 API 透明切换 Higress AI Gateway
P0 自建 GPU 推理集群：高性能 LLM 本地推理，数据不出集群 ACK 灵骏裸金属 H20 SGLang/vLLM
P0 云端 API Fallback：自建推理不可用时自动切换，保障业务连续性 百炼 API
P1 AI 全链路可观测：从网关到 GPU 卡的全栈监控（Token 统计、延迟/吞吐、GPU 利用率/温度） Prometheus Grafana DCGM Loki
P1 AI 内容合规：全量 Prompt/Response 审计，满足监管要求，NAS 持久化 Higress 审计日志 Loki NAS
P1 K8s 统一 GPU 调度：ECS 与裸金属统一资源视图，弹性扩缩 ACK GPU Operator LWS Controller
P2 跨机高性能互联：大模型分布式推理 Tensor Parallel 通信 RDMA RoCE Multus CNI Mellanox ConnectX
P2 多业务 App 资源隔离：不同 App 不互相阻塞 Higress 限流 SGLang 硬隔离

关键设计原则
1
统一网关 所有 LLM 请求经 Higress 管控，业务 App 无需感知后端推理引擎变更
2
混合推理双轨 自建 GPU 主力 + 百炼 API Fallback，Higress 健康检查自动熔断切换
3
全链路可观测 网关 → 推理引擎 → GPU 卡级指标一体化，不散装拼凑
4
内容合规 全量 Prompt/Response 审计，NAS 持久化，满足国内监管要求
5
高性能互联（按需启用） RDMA RoCE 保障跨机 Tensor Parallel 性能，≤72B 模型单机 TP=8 无需跨机

⚔️ 06  与 AWS 原方案对比
能力层✅ 阿里云方案优势❌ AWS 原方案局限
AI 网关统一入口 Higress 一个入口管控所有 LLM 流量；AI 路由 + 熔断降级 + 灰度发布一体化 业务侧自行对接多推理 Endpoint，运维复杂度高
AI 可观测 网关 → GPU 卡全链路一体化监控（Token、延迟/吞吐、GPU 利用率/温度） 各层监控散装拼凑，难以形成全链路视图
AI 内容合规 全量 Prompt/Response 持久化审计，满足国内监管；按 App 维度访问控制 合规审计需额外开发，难满足国内监管要求
K8s 统一 GPU 调度 ACK 一套集群统一纳管 ECS + 灵骏裸金属 + 百炼 API，资源视图一体化 GPU 与业务调度分离，资源视图割裂
混合推理双轨 自建 GPU + 百炼 Fallback，健康检查自动熔断，高可用兜底 Fallback 逻辑需业务侧自行实现
灵骏 AI 扩展内核 kernel 6.8.0-aiext，针对 AI 工作负载专项优化，推理性能领先 通用 kernel，无 AI 专项优化
💡 07  方案优化建议
#优先级建议说明
1高Higress Gateway HA1 副本是单点故障，扩至 2-3 副本 + 反亲和 + SLB 四层负载
2高Fallback 明确触发条件健康检查 + 限流降级 + 熔断保护三档配置，否则"双轨"实为冷备
3高灵骏 + Ubuntu 兼容性MOFED 版本与 aiext kernel 需锁版本；NCCL_IB_HCA 等参数需显式配置
4高跨机 TP 必要性评估H20 单机 1,128GB，≤72B 模型无需跨机；多副本 DP 通常 ROI 更高
5中两套 Prometheus 收敛Higress 自带 Prometheus 改走 ServiceMonitor，统一为集群级一套
6中Prompt 缓存层Higress 已有 Redis，审计类重复 prompt 场景可节省 30-50% Token
7中多 App 资源隔离早期软隔离（Higress 限流配额），业务量大后考虑 SGLang 硬隔离
8中审计日志容量规划全量存储约每月几百 GB，需明确合规要求是全量还是元数据+异常全量
9低GPU 成本归集DCGM + 业务 label，可算出单次会话 GPU 成本，对接业务定价
10低灰度发布机制Higress 按权重灰度，预留标准 SOP：5% → 观察 30min → 全量
11低RDMA DP nodeSelector当前 4 Pod 包含 ECS 节点，浪费资源，应仅调度到 GPU 节点
