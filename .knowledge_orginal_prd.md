# Cloud & AI Knowledge Flywheel

> 基于 Qoder Skills 驱动的 Cloud & AI 知识飞轮系统

---

## 原始需求

### 背景

作为阿里云 SA（解决方案架构师），日常工作中持续产生大量技术知识：产品理解、竞品分析、客户方案、AI 领域洞察等。这些知识散落在对话、笔记、脑海中，缺乏系统化沉淀和复用机制。

### 核心诉求

1. **构建一个可迭代的知识飞轮**：创建知识 → 校验知识 → 更新知识，形成正循环
2. **知识以 Markdown 为主**，讲究通用性，后续需托管到 GitHub
3. **基于本地 Mac 电脑运行**，每周同步 GitHub 即可
4. **利用 Qoder Skill 机制驱动**，让 AI 参与知识的消费、生产和质控

### 覆盖场景

| 场景 | 说明 | 示例 |
|------|------|------|
| **AI 领域知识** | LLM、VLM、AI 可观测、AI 工程等跨厂商技术知识 | RAG 原理、GPU 选型、Harness 架构模式 |
| **整体竞争分析** | 阿里云与 AWS、火山引擎等云厂商对比 | 阿里云 vs AWS 全产品线对比 |
| **单产品知识** | 每个产品的原理、边界、误解、踩坑记录 | Qoder 产品定位与适用边界 |
| **产品对比分析** | 同品类产品横向对比 | Qoder vs Kiro |
| **行业解决方案** | 针对特定客群的方案分析 | MNC、出海、短剧、IPC 摄像头客群 |

### 关键约束

- 报价功能不做（价格信息时效性强，不适合静态沉淀）
- 所有事实性内容必须引用可信信息源，不能猜测
- 需控制 Token 消耗，Skill 执行时按需加载而非全量读取
- 知识图谱放二期（知识量 >30 篇后启动）

### 可信信息源

| 类别 | 来源 |
|------|------|
| AI/大模型 | arXiv、Anthropic（含 Engineering Blog）、OpenAI、Google DeepMind |
| 云厂商 | 阿里云官方文档、AWS 官方文档、火山引擎文档 |
| 行业分析 | Gartner、IDC、Forrester |
| 参考架构 | [Anthropic Harness Design](https://www.anthropic.com/engineering/harness-design-long-running-apps)、AWS Well-Architected、阿里云架构中心 |

## 方案设计

### 飞轮运转机制

```mermaid
graph LR
    A["用户提问"] --> B["Expert Skill 回答"]
    B --> C["发现知识缺口"]
    C --> D["Knowledge Miner Skill 提炼"]
    D --> E["Validator Skill 校验"]
    E --> F["知识入库"]
    F --> B
```

**知识流转路径**：

```
inbox/（随手记）→ staging/（Miner 结构化）→ knowledge/（Validator 审批）→ archive/（inbox 归档）
```

| 阶段 | 目录 | 触发 Skill | 说明 |
|------|------|-----------|------|
| 投递 | `inbox/` | — | 用户随手记录原始素材 |
| 提炼 | `staging/` | Knowledge Miner | 结构化为标准模板格式 |
| 校验 | `staging/` → `knowledge/` | Validator | 校验通过后移入正式库 |
| 消费 | `knowledge/` | Expert | 基于正式知识回答问题 |

### 知识层次体系

| 层次 | 目录 | 核心问题 | 示例 |
|------|------|---------|------|
| 道 | `ai/` | "这个技术是什么？怎么选？" | RAG 原理、GPU 选型 |
| 点 | `product/` | "这个产品怎么用？边界在哪？" | 百炼 RAG、PAI GPU |
| 线 | `product-analysis/` | "A 和 B 怎么比？" | 百炼 vs Bedrock |
| 面 | `competitive-analysis/` | "两家厂商整体差异？" | 阿里云 vs AWS |
| 体 | `solutions/` | "这类客户怎么服务？" | 短剧方案 |

> `ai/` 回答"技术是什么"，`product/` 回答"产品怎么用"。

## Skill 体系（3 个）

| Skill | 文件 | 职责 | 触发条件 |
|-------|------|------|----------|
| **Expert Skill** | `.skills/expert.md` | 基于知识库回答问题，标注知识缺口 | 用户提出 Cloud & AI 相关问题 |
| **Knowledge Miner Skill** | `.skills/knowledge-miner.md` | 从 inbox 和可信源提炼结构化知识 | 用户要求"提炼知识"、"整理笔记" |
| **Validator Skill** | `.skills/validator.md` | 入库前校验 + 定期巡检 + changelog 生成 | 用户要求"校验"、"审核"、"巡检" |

### Expert Skill

- **输入**：用户提问 + `knowledge/index.md` + 目标文档
- **输出**：结构化回答 + 知识缺口标记
- **核心行为**：
  - 优先读 `index.md` 索引定位，再读目标文档（按需加载）
  - 知识不存在标注 `[📭 知识缺口]`，建议触发 Miner
  - 知识超 90 天未更新标注 `[🔴 可能过期]`
  - 回答中引用知识文档路径，方便溯源

### Knowledge Miner Skill

- **输入**：`inbox/` 原始素材 + 可信信息源
- **输出**：`staging/` 结构化草稿
- **核心行为**：
  - 读取 `inbox/` 中指定日期的文件
  - 自动识别知识分类（ai / product / competitive / product-analysis / solution）
  - 只读目标分类的 `_template.md`，生成结构化文档
  - 输出到 `staging/`，完成后将 inbox 移至 `archive/`
  - 所有事实性内容必须标注来源

### Validator Skill

- **输入**：`staging/` 或 `knowledge/`
- **输出**：校验报告 + changelog 更新
- **两种模式**：
  - **Gate 模式**：`staging/` → `knowledge/` 入库审批
  - **Patrol 模式**：定期扫描 `knowledge/` 健康度
- **校验清单**：
  - [ ] 文档是否符合对应 `_template.md` 格式
  - [ ] 所有事实性陈述是否附带引用来源
  - [ ] 引用来源是否在 `trusted-sources.md` 中注册
  - [ ] 未注册来源是否已标注 `[⚠️ 未验证来源]`
  - [ ] 价格/规格类数据是否标注查询日期
  - [ ] 文档是否超过 90 天未更新（标注 🔴）
  - [ ] 是否存在未标注的主观猜测
  - [ ] 架构图是否使用 Mermaid 语法
- **校验通过后**：更新 `index.md` + 追加 `changelog`

## Token 优化策略

| # | 规则 | 效果 |
|---|------|------|
| 1 | **索引先行**：新增 `index.md`，所有 Skill 先读索引再取文件 | Token 降 ~85% |
| 2 | **懒加载**：Skill 中明确读取顺序和条件，永远不全量加载 | 避免无效读取 |
| 3 | **文档摘要块**：每篇文档顶部嵌入 SUMMARY 块 | 快速判断是否需读全文 |
| 4 | **模板精简**：公共结构（Frontmatter、参考资料、Changelog）Skill 内置理解 | 模板体积降 ~40% |

### Skill 读取策略

```
Expert Skill:
  1. 必读 index.md（~200行）
  2. 根据问题匹配，只读 1~2 个目标文档
  3. 不读模板文件

Knowledge Miner:
  1. 只读 inbox/ 中指定文件
  2. 只读目标分类的 _template.md（1个）
  3. 入库后追加更新 index.md

Validator:
  1. 只读 staging/ 中待校验的文件
  2. 只读对应 _template.md 做格式比对（1个）
  3. 通过后更新 index.md + changelog
```

### Zsh 快捷命令（加入 ~/.zshrc）

```bash
# ===== Cloud AI Knowledge Flywheel =====
export KB_HOME="$HOME/knowledge-base/cloud-ai-knowledge-base"

alias kb="cd $KB_HOME"

kbnew() {
  local today=$(date +%Y-%m-%d)
  local file="$KB_HOME/inbox/$today.md"
  if [ ! -f "$file" ]; then
    cat > "$file" << EOF
# $today 随手记

## 今天了解到
- 

## 待深入研究
- 

## 链接收藏
- 
EOF
    echo "📝 已创建：$file"
  fi
  open "$file"
}

kbstatus() {
  echo "📥 inbox 待处理："
  ls -1 "$KB_HOME/inbox/"*.md 2>/dev/null | wc -l | xargs echo "   文件数:"
  echo "🔄 staging 待校验："
  find "$KB_HOME/staging" -name "*.md" 2>/dev/null | wc -l | xargs echo "   文件数:"
  echo "✅ knowledge 正式知识："
  find "$KB_HOME/knowledge" -name "*.md" ! -name "_template.md" ! -name "index.md" 2>/dev/null | wc -l | xargs echo "   文件数:"
  echo "📦 archive 已归档："
  ls -1 "$KB_HOME/archive/"*.md 2>/dev/null | wc -l | xargs echo "   文件数:"
}

kbsync() {
  cd "$KB_HOME"
  git add .
  git commit -m "knowledge: weekly sync $(date +%Y-%m-%d)"
  git push origin main
  echo "🚀 已同步到 GitHub"
}
```
