## 产品原理解析

### 一句话定位

百炼是阿里云的 MaaS（Model-as-a-Service）平台，提供多地域统一的模型调用接入能力，兑现为 OpenAI 兼容 API。

### 地域与接入点

| 地域 | 接入点 | 适用范围 |
|------|--------|--------|
| 中国大陆（默认）| `dashscope.aliyuncs.com` | 大陆站 UID，Qwen / DeepSeek-V4 / Wan 系列 |
| 新加坡国际 | `dashscope-intl.aliyuncs.com` | 国际站 UID，Qwen / DeepSeek-V4 / DeepSeek-V3.2 |
| 美国 | `dashscope-us.aliyuncs.com` | 国际站 UID，主要面向 DeepSeek-V4-* 与 qwen-plus-us / qwen-flash-us |

> **重要**：大陆站账号体系与国际站独立，同一款物理人在两边注册会产生两个独立 UID。这是多账号扩 TPM 方案可行的前提之一。

### 限流机制（核心限制）

**限流颗粒度：阿里云账号（UID）级别**，覆盖该账号下所有的：
- RAM 用户
- 工作空间
- API Key

同一账号下创建 N 个 API Key 不扩展额度，额度是全局共享的。

**限流维度**：

| 维度 | 说明 | 触发后表现 |
|------|------|---------|
| RPM | 每分钟请求数 | HTTP 429 + "Requests rate limit exceeded" |
| TPM | 每分钟 Token 数（入+出）| HTTP 429 + "Allocated quota exceeded" |
| RPS / TPS | 秒级平滑保护（= RPM/60 / TPM/60）| HTTP 429 + "Request rate increased too quickly" |

**恢复特性**：触发限流后通常 1 分钟内自动恢复，不需人工干预。

### 文档限流与实测限流的差异⚠️

百炼官方文档发布的 RPM/TPM 是「理论上限」，实测会明显打折扣，以 deepseek-v4-flash 思考模式为例：

| 项 | 文档值 | 实测值 | 差异 |
|----|--------|--------|------|
| RPM（成功）| 15,000 | ~300-500 | 仅为文档值 1/30 |
| RPM（到达服务端，含 429）| - | ~1,500 | - |
| TPM | 1,200,000 | 未接近（RPM 先被限）| - |

**推测原因**：
- 思考模型算力消耗高，服务端对该类模型设了更严的软限制
- 账号默认额度可能远低于控制台「限流提额」页面公示的额度
- 地域、账号等级、突发保护有叠加打折

**生产原则**：生产部署前必须用与生产一致的负载（模型+并发+消息长度）跑一次端到端压测，不要盲信文档额度。

## 多账号扩 TPM 方案

### 适用场景

单个 UID 额度无法满足业务吞吐需求、且官方提额不够或审批号隆占时，可通过注册多个独立阿里云账号，在业务层路由以获得近似 N 倍额度。

### 实测验证（2 UID 跑 deepseek-v4-flash）

**环境**：大陆站 UID + 新加坡国际站 UID，思考模式默认开启，500 总并发。

| 指标 | 单 UID 500 并发 | 2 UID 250×2 并发 | 倍数 |
|------|---------------:|-------------:|----:|
| 60s 窗口 TPM | 217K | **1,840K** | ×8.5 |
| 429 限流 | 1,449 | **0** | 完全消除 |
| 成功请求 | 346 | 3,359 | ×9.7 |
| 平均延迟 | 12.0s | 9.3s | ⬇️ 22% |

**结论**：2 个独立 UID 实测 TPM **突破单 UID 文档额度（1.2M）达到 1.84M+**，且零限流。各账号分摄均衡（13:47% vs 47:53%，几乎对半分）。

### 架构需求

```
业务应用
   │
账号调度层（需业务自建）
   ├─ 账号池（N 个独立 UID 的 API Key）
   ├─ 加权轮询 / 最少负载调度
   ├─ 429 检测 → 账号熔断 60-70s
   ├─ 独立 httpx 连接池 + client 复用
   └─ 降级策略（备用模型 / Batch API）
   │
   ▼
百炼多个 UID
```

**关键点**：
1. **N 个独立 UID**（大陆站 vs 国际站、或多个大陆站账号）——同账号下多 Key 无效
2. **base_url 与地域匹配**，法事账号使用错误 base_url 将返回鉴权错误
3. **客户端资源需要优化**：httpx 连接池 max_connections ≥ 600、AsyncOpenAI 复用、ulimit 调高
4. **总并发控制在 N × 单账号甜点**，避免客户端自我 DDoS

### 多账号 vs 其他方案决策框架

| 场景 | 首选方案 | 原因 |
|------|---------|------|
| 偶尔超额 | 控制台 "限流提额" | 运维成本最低 |
| 稳定高并发 | 多账号池化 | 提额存在上限，多账号可线性扩展 |
| 离线批处理 | Batch API | 不受实时限流，成本更低 |
| 超高吞吐+低延迟 | 自托管 GPU（火山/灵骏）| 多账号运维成本超过自建 |

### 隐形成本

| 项 | 说明 |
|----|------|
| 多份账单 | N 个账号独立计费，需统一充值 & 欠费告警 |
| API Key 轮换 | N 倍密钥轮换运维负担 |
| 非均匀限流 | 某个账号可能因突发流量先被打满 |
| 合规灰色地带 | 官方未明确禁止，但「限流提额」才是官方推荐路径 |

## 适用边界分析

### ✅ 适用场景

| 场景 | 说明 | 典型客户/案例 |
|------|------|--------------|
| 多模型联调 | Qwen + DeepSeek + Wan 统一接入 | 企业智能助手 |
| 快速 POC | 免部署、按 token 计费 | 创创业公司 MVP |
| 多地域调用 | 大陆/新加坡/美国三点均提供 OpenAI 兼容 API | 出海业务 |

### ❌ 不适用场景

| 场景 | 不适用原因 | 替代方案 |
|------|-----------|----------|
| 超高 RPM 实时场景 | 单 UID 限流上限明显 | 多账号池化 + 火山/灵骏 GPU 自托管 |
| 严格数据主权隔离 | 数据可能进入指定地域之外的混合调度 | Global 部署模式 / VPC 内网部署集群 |
| 超低延迟（<200ms）| MaaS 平均延迟 > 1s | 业务侧缓存 + 小模型本地部署 |

### ⚠️ 常见误解

| 误解 | 事实 |
|------|------|
| "创建多个 API Key 就能扩额" | 限流按 UID 计，同账号多 Key 共享额度 |
| "文档写 RPM=15000 我就能跑 15000" | 实测可能只能跑到 1/30，生产前必需压测 |
| "大陆账号和国际站是一个账号" | 两者是独立账号体系，限流互不影响 |
| "提高客户端并发就能提高吞吐" | 存在拥塞拐点，过高并发反而崩塌 |

## 关键配置与最佳实践

### 限流发生后的处理

1. **判别错误类型**：
   - "Requests rate limit exceeded" → RPM 超限，限流调用频率
   - "Allocated quota exceeded" → TPM 超限，压缩输入输出长度
   - "Request rate increased too quickly" → 秒级保护，平滑调用曲线
2. **调用侧退避**：指数退避 + 拖全、请求队列缓冲
3. **备用模型**：qwen-plus 被限 → 自动切 qwen-plus-2025-07-14 或 qwen-flash
4. **多账号池化**（本页上述方案）
